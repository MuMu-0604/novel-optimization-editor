# Novel Optimization Editor

一个用于长篇小说、网文、剧本式小说和多章节虚构作品的 AI 编辑 Skill。它的目标不是让 AI 自由续写，而是让 AI 在作者授权范围内进行情节诊断、逐字分析、结构优化、场景改写和连续性校验。

本 Skill 支持任意大小的小说原文。原文大小不会成为处理限制；如果全文无法一次进入上下文，Skill 会要求按章节或场景分块读取，并通过覆盖清单追踪已读取、已索引和已检查的范围。

## 核心能力

- 小说全文读取与分块索引
- Story Bible 剧情数据库建立
- 主线、支线、人物动机、因果链诊断
- 事件时间线、叙事顺序、信息揭示顺序控制
- 伏笔埋设与回收检查
- 人物状态、关系变化、知识边界校验
- 逐字逐句细节分析
- 作者授权范围内的场景改写
- 修改风险评估与改后同步
- 防止 AI 乱编、乱改设定、乱动结局

## 文件结构

```text
novel-optimization-editor/
  SKILL.md
  README.md
  agents/
    openai.yaml
  references/
    story-bible-template.md
    pressure-tests.md
```

说明：

- `SKILL.md`：核心 Skill 规则，供支持 Skill 的 Agent 读取。
- `README.md`：本说明文档，供作者或开发者了解使用方法。
- `agents/openai.yaml`：OpenAI/Codex 风格 Skill 列表元数据。
- `references/story-bible-template.md`：剧情数据库模板。
- `references/pressure-tests.md`：测试 Agent 是否遵守规则的压力场景。

## 安装方式

### 方式一：安装到支持 Skill 的 Agent

将整个文件夹放入目标 Agent 的 skills 目录中，例如：

```text
skills/
  novel-optimization-editor/
    SKILL.md
    agents/
    references/
```

然后在对话中提出类似请求：

```text
请使用 novel-optimization-editor 帮我优化这部长篇小说。
```

### 方式二：用于不支持 Skill 目录的 Agent

如果目标 Agent 不支持 Skill 文件夹，可以把 `SKILL.md` 的内容复制到 system prompt、project instruction 或自定义指令中。

如果 Agent 支持附加参考文档，再同时提供：

```text
references/story-bible-template.md
references/pressure-tests.md
```

## 推荐运行流程

### 第一步：建立作者控制面板

开始优化前，先让 AI 记录作者意图和权限边界：

```text
作品类型：
目标读者：
叙事视角：
文风目标：
读者核心期待：
不可更改设定：
预定结局：
允许修改等级：
允许新增内容：
禁止改动内容：
是否允许调整事件顺序：
是否允许调整信息揭示顺序：
输出偏好：诊断 / 方案 / 直接改写 / 修改记录
```

这一环节用于防止 AI 把“优化”理解成“重写整本书”。

### 第二步：读取小说原文

小说原文大小不限。根据实际上下文容量选择读取方式：

```text
全文可一次放入上下文：直接读取，但仍需建立索引。
全文过长：按章节读取。
单章过长：按场景继续拆分。
```

每个分块读取后，AI 应输出：

```text
分块编号：
覆盖章节/场景：
已确认事件：
人物状态变化：
新增或变化设定：
时间线条目：
信息揭示条目：
伏笔埋设/回收：
矛盾或不确定信息：
推断出的禁改内容：
```

### 第三步：维护覆盖清单

AI 必须记录全文读取进度：

```text
作品 ID：
已知总章节/文件：
已收到：
已处理：
已索引：
未检查缺口：
当前覆盖状态：部分 / 完整
是否可以进行全书诊断：是 / 否
```

如果只读取了部分章节，AI 不能声称已经完成全书分析。

### 第四步：建立 Story Bible

使用 `references/story-bible-template.md` 建立剧情数据库，包括：

```text
章节索引
正典事实表
真实时间线
叙事呈现顺序
信息揭示顺序
人物状态表
伏笔账本
未解问题表
修改日志
```

这是后续防止乱编、乱改顺序、乱动伏笔的核心依据。

### 第五步：诊断问题

诊断阶段只找问题，不直接改写。每个问题应包含：

```text
问题编号：
位置：
原文依据：
问题类型：
为什么影响阅读：
修改等级：
最小修改方案：
强化修改方案：
风险：
是否需要作者确认：
```

常见问题类型：

```text
主线不清
人物动机不足
因果断裂
节奏拖沓
冲突不足
场景功能重复
信息提前暴露
伏笔缺失
伏笔未回收
人物转变过快
对白解释感过强
场景没有转折
```

### 第六步：按权限等级修改

Skill 使用 5 级修改权限：

```text
L1 语言级：
病句、重复、弱词、语序、节奏、清晰度。可直接修改。

L2 场景级：
动作、对白、心理、环境、微冲突，不改变剧情结果。可直接修改。

L3 情节逻辑级：
补动机、修因果、加过渡、删合重复场景。需要方案，除非作者预授权。

L4 结构级：
章节顺序、信息揭示顺序、伏笔位置、高潮重排、人物线重构。必须作者确认。

L5 核心设定级：
结局、主角身份、主要关系结局、世界观底层规则、生死结局。默认禁止，除非作者明确点名授权。
```

### 第七步：改写前做风险判断

L3 以上改动必须判断风险：

```text
风险等级：低 / 中 / 高 / 禁止
影响范围：语言 / 场景 / 情节 / 结构 / 世界观
影响记录：时间线 / 伏笔 / 人物状态 / 设定 / 后文
回滚难度：
是否需要作者确认：
```

涉及时间线、信息揭示、伏笔、关系推进的改动，必须先输出影响报告。

### 第八步：执行改写

改写时只处理授权范围。改写后输出：

```text
改写文本：

修改日志：
- 位置：
- 改动：
- 理由：
- 修改等级：
- 影响记录：

一致性检查：
- 是否改变剧情事实：
- 是否改变事件顺序：
- 是否改变信息揭示顺序：
- 是否改变人物知识边界：
- 是否影响伏笔：
- 是否新增未授权内容：
- 是否需要作者复核：
```

### 第九步：同步剧情数据库

改写后必须同步或列出需要同步的记录：

```text
章节索引：
正典事实表：
真实时间线：
叙事呈现顺序：
信息揭示顺序：
人物状态表：
伏笔账本：
未解问题：
修改日志：
```

## 常用提示词

### 读取整本小说

```text
请使用 novel-optimization-editor 进入 Intake Mode。
我会按章节提供小说原文。
当前阶段只读取、索引、建立 Story Bible，不要优化、续写或改写。
如果信息不足，请标注“资料不足”，不要自行补全。
```

### 逐字分析

```text
请进入 Line Edit Mode，对以下片段逐字逐句分析。
不要直接整体润色。
请指出具体问题字词、问题类型、为什么影响阅读、修改建议、连续性风险。
```

### 情节诊断

```text
请进入 Diagnosis Mode。
基于已建立的 Story Bible，诊断当前章节的情节、人物动机、因果链、节奏、伏笔和信息揭示问题。
先诊断，不要直接改写。
```

### 授权改写

```text
我授权你进行 L1-L2 修改。
L3 以上只提出方案，不要直接改。
请优化以下场景，并在改写后输出修改日志和一致性检查。
```

### 结构调整

```text
请进入 Structure Mode。
我想调整章节顺序或信息揭示顺序。
请先输出 Sequence Impact Report，说明对时间线、人物知识、伏笔和后文的影响。
未经我确认，不要改写正文。
```

## 适用场景

- 长篇小说全书诊断
- 网文章节优化
- 出版稿结构编辑
- 人物动机修补
- 伏笔检查与回收
- 大纲重排
- 逐字逐句精修
- 防止 AI 在优化中乱编剧情
- 多 Agent 协作编辑小说

## 不适用场景

- 让 AI 完全替作者原创一部长篇小说
- 无视作者设定进行自由续写
- 在没有原文依据时强行补完设定
- 在只读取部分章节时做全书结论

## 设计原则

这个 Skill 的核心不是“让 AI 写得更多”，而是让 AI：

```text
先读取
再建档
再诊断
再分级授权
再改写
最后校验和同步
```

它把 AI 从自由发挥的写手，限制为有记录、有边界、可追踪、可回滚的小说编辑。
